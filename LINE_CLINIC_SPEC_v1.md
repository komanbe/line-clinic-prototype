## 1. プロジェクト概要

LINE公式アカウント上で完結するオンライン診療予約・ビデオ通話・薬配送システム。  
エニキュアをベースに、以下の差別化を行う:

| 観点 | エニキュア | 本システム（改善） |
|------|----------|----------------|
| ビデオ通話 | Zoom（外部遷移） | **LINE内WebRTCビデオ通話**（LIFF内完結） |
| 医師選択 | 空き時間ベース（医師が選びにくい） | **ニーズ別 + 医師指名の両方対応** |
| 入口の分岐 | なし（全員同じフロー） | **ニーズ別分岐**（薬のみ / 相談 / カウンセリング） |
| 診療カテゴリ | 医師 / 心理士 / PSW が混在気味 | **明確にタブ分離 + 専門領域タグ** |
| アバター | なし | **Vアバター対応**（患者側オプション） |
| 画面共有 | なし | **書類共有機能**（診断書確認等） |
| 悩み相談掲示板 | あり（症状悪化リスク） | **廃止 or 医療監修付きQ&A** |
| ブランディング | IT効率化寄り | **誠実な医療 + ニーズ別対応** |

## 2. 技術スタック

### 2.1 全体構成

| レイヤー | 技術スタック | 役割 |
|---------|------------|------|
| フロントエンド | **LIFF (Vite + React)** | 予約UI・問診・ビデオ通話・マイページ |
| 認証 (LINE) | LINE Login Channel | LINE userId 取得 |
| バックエンド | **Cloudflare Workers (Hono)** | API・認証・セッション管理 |
| データベース | **Cloudflare D1 (SQLite)** | 予約・患者・医師・問診データ |
| ストレージ | **Cloudflare R2** | 保険証画像・診断書PDF |
| ビデオ通話 | **Cloudflare Calls (WebRTC)** or **Twilio Video** | LINE内ビデオ通話 |
| 配信エンジン | **LINE Harness OSS** ステップ配信 | リマインド・フォローアップ配信 |
| 管理画面 | **Next.js 15** (LINE Harness既存拡張) | 予約管理・医師シフト・売上管理 |
| 決済 | **Stripe** | クレジットカード・保険診療決済 |

### 2.2 UTokyo SSOアンケートプロジェクトとの共通基盤

| 共通モジュール | 内容 | 備考 |
|-------------|------|------|
| LINE Harness OSS | BAN検知・フェイルオーバー・セグメント配信 | そのまま共用 |
| Cloudflare Workers + D1 | バックエンド基盤 | Workers プロジェクトは分離、D1スキーマは別 |
| LIFF + Vite + React | フロントエンド基盤 | 共通UIコンポーネントライブラリを切り出し可能 |
| LINE Login 認証フロー | liff.getAccessToken → Workers検証 | 認証ミドルウェアを共通npm化 |
| 途中保存パターン | セクション単位保存 + sessionStorage | 問診フォームで同じパターン適用 |
| 配信テンプレート | リマインド・セグメント配信 | LINE Harnessの共通機能 |

**分離すべき部分:**
- D1スキーマ（患者/医師/予約 vs アンケート/回答）
- Workersプロジェクト（別デプロイ）
- LIFF ID（別チャネル）
- UTokyo SSO認証（診療システムでは不要）

## 4. ビデオ通話（LINE内WebRTC）

### 4.1 技術選定

エニキュアはZoomを使用（外部アプリ遷移が必要 → UX悪い）。  
本システムは**LIFF内でWebRTCビデオ通話を実現**する。

| 選択肢 | メリット | デメリット | 推奨 |
|--------|---------|-----------|------|
| **Cloudflare Calls** | Workersと統合、低コスト、SFU対応 | 比較的新しいサービス | ★推奨 |
| **Twilio Video** | 実績豊富、SDK充実 | コスト高め（$0.004/分/参加者） | 次点 |
| **自前WebRTC (PeerJS等)** | 無料 | TURN/STUNサーバ運用必要、品質保証難しい | 非推奨 |
| **LINE公式のビデオ通話** | LINE native | APIが公開されていない、制御不可 | 不可 |

### 4.2 実装方式（Cloudflare Calls推奨）

```
患者(LIFF)                   Cloudflare Calls SFU             医師(管理画面)
    │                              │                              │
    │─── JOIN room ───────────────→│                              │
    │                              │←──── JOIN room ──────────────│
    │←── Media Stream ────────────→│←───── Media Stream ─────────→│
    │                              │                              │
    │   [画面共有]                  │                              │
    │─── Screen Share ────────────→│────── Screen Share ──────────→│
    │                              │                              │
    │   [書類送信]                  │                              │
    │─── File via DataChannel ────→│                              │
```

**LIFF内での実現性:**
- LIFF v2はLINE内ブラウザ（WebView）で動作
- iOS/Android WebViewはWebRTC対応済み（getUserMedia, RTCPeerConnection）
- **注意:** iOS LINE内ブラウザでカメラ・マイクパーミッションが必要 → LIFF SDKの`liff.permission`で事前取得
- 画面共有: モバイルでは制約あり（iOS非対応の可能性）→ 代替として**書類画像送信機能**を実装

## 5. Vアバター機能

### 5.1 概要

患者が顔出しに抵抗がある場合、**自分の映像をVアバターに置き換える**オプション。  
メンタルヘルス領域では顔出し抵抗のある患者が多く、差別化要素になる。

### 5.2 技術選定

| 選択肢 | 概要 | 実現性 |
|--------|------|--------|
| **MediaPipe Face Mesh + Three.js** | ブラウザ内で顔認識→3Dアバター描画 | ★推奨（LIFF内で動作可能） |
| **Kalidokit + VRM** | MediaPipeベースのVRMアバター制御 | 良い（VRMモデル読み込み） |
| **TensorFlow.js Face Landmarks** | 顔のランドマーク検出 | 軽量だがアバター描画は別途必要 |
| **外部SDK (Ready Player Me等)** | フルスタックアバターSDK | 重い、コスト発生 |

### 5.3 実装方針（MediaPipe + Three.js）

```
カメラ映像 → MediaPipe Face Mesh（顔ランドマーク52点検出）
    ↓
顔の向き・表情パラメータ抽出（口の開閉、目の開閉、首の傾き等）
    ↓
Three.js で VRMアバターに適用
    ↓
Canvas → MediaStream → WebRTC送信
```

**パフォーマンス:**
- MediaPipe Face MeshはWASM/GPU対応で30fps可能
- モバイルでも動作するが、低スペック端末では15fps程度
- アバター使用時はカメラ映像は送信せず、Canvas映像のみ送信（プライバシー保護）

### 5.4 提供アバター

- 初期リリース: 5種類程度のプリセットアバター（シンプルなアニメ調）
- 将来: カスタマイズ（髪型、色、アクセサリ等）

### 5.5 制約・リスク

| リスク | 対策 |
|--------|------|
| モバイルでのパフォーマンス | アバター品質を3段階で選択可能に（高/中/低） |
| iOS WebViewの制約 | MediaPipe WASM版で対応、GPU版はフォールバック |
| 通信量増加 | アバター映像はCanvas経由で解像度制御可能 |
| 医師が患者の表情を読めない | アバターが表情を反映する設計（口・目・眉） |

**結論: V1では「あると嬉しい」レベル（P2）。ビデオ通話が安定してからPhase 2で実装推奨。**

## 6. 医師・スタッフ管理

### 6.1 登録する職種

| 職種 | 役割 | 枠の長さ |
|------|------|---------|
| 医師（精神科・心療内科） | 診察・処方・診断書発行 | 10-20分 |
| 臨床心理士 | カウンセリング | 30分 |
| PSW（精神保健福祉士） | 生活・社会資源の相談 | 30分 |

### 6.2 医師プロフィール（改善版）

エニキュアは名前と経歴のみ → **専門タグ・得意分野を追加**

| フィールド | 例 |
|-----------|---|
| 氏名 | 田中太郎 |

| 対応枠 | 朝/昼/夜 |

### 6.3 シフト管理

**エニキュアのスキーム（成果報酬型）の採用:**
- 医師が自分で空き枠を登録
- 予約が入った分のみ報酬発生 → 無駄な人件費なし
- 枠が増えれば無限にスケール可能

```

│                                  │
│ 4/17(金)  08:00 - 12:00  [登録]  │
```

### 6.4 医師のコード・品質管理（matsu先生の意見反映）

> 「登録する医師に対し、我々のコードを守っていただくことも差別化には大事」

- 登録時にコード・ガイドラインへの同意必須
- NG行為の明確化（不適切な処方、ゆうメンタル的な量産診療 等）
- 患者からの評価フィードバック（非公開、運営確認用）

### 8.1 主要テーブル

```sql
-- 患者
CREATE TABLE patients (
  id TEXT PRIMARY KEY,
  line_user_id TEXT UNIQUE NOT NULL,
  name TEXT NOT NULL,
  name_kana TEXT NOT NULL,
  birth_date TEXT NOT NULL,
  gender TEXT NOT NULL,
  postal_code TEXT,
  address TEXT,
  building TEXT,
  phone TEXT,
  email TEXT,
  hospital_name TEXT,
  insurance_card_front_key TEXT,  -- R2 object key
  insurance_card_back_key TEXT,
  created_at TEXT DEFAULT (datetime('now')),
  updated_at TEXT DEFAULT (datetime('now'))
);

-- 医師・スタッフ
CREATE TABLE doctors (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  name_kana TEXT NOT NULL,
  role TEXT NOT NULL,  -- 'doctor' | 'psychologist' | 'psw'
  photo_url TEXT,
  affiliation TEXT,
  bio TEXT,
  specialties TEXT,  -- JSON array: ["うつ", "ADHD"]
  consultation_tags TEXT,  -- JSON array: ["ビジネス", "学生"]
  message TEXT,
  is_active INTEGER DEFAULT 1,
  created_at TEXT DEFAULT (datetime('now'))
);

-- シフト（空き枠）
CREATE TABLE doctor_shifts (
  id TEXT PRIMARY KEY,
  doctor_id TEXT NOT NULL REFERENCES doctors(id),
  date TEXT NOT NULL,
  start_time TEXT NOT NULL,
  end_time TEXT NOT NULL,
  slot_minutes INTEGER NOT NULL DEFAULT 20,
  is_available INTEGER DEFAULT 1,
  created_at TEXT DEFAULT (datetime('now'))
);

-- 予約
CREATE TABLE reservations (
  id TEXT PRIMARY KEY,
  patient_id TEXT NOT NULL REFERENCES patients(id),
  doctor_id TEXT NOT NULL REFERENCES doctors(id),
  shift_id TEXT NOT NULL REFERENCES doctor_shifts(id),
  consultation_type TEXT NOT NULL,  -- 'medication' | 'consultation' | 'counseling' | 'document'
  visit_type TEXT NOT NULL,  -- 'first' | 'revisit'
  date TEXT NOT NULL,
  start_time TEXT NOT NULL,
  end_time TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'confirmed',  -- 'confirmed' | 'completed' | 'cancelled' | 'no_show'
  video_room_id TEXT,
  fee_amount INTEGER,
  payment_status TEXT DEFAULT 'pending',  -- 'pending' | 'paid' | 'refunded'
  stripe_payment_id TEXT,
  created_at TEXT DEFAULT (datetime('now')),
  updated_at TEXT DEFAULT (datetime('now'))
);

-- 問診回答
CREATE TABLE questionnaires (
  id TEXT PRIMARY KEY,
  reservation_id TEXT NOT NULL REFERENCES reservations(id),
  answers TEXT NOT NULL,  -- JSON
  completed INTEGER DEFAULT 0,
  section_index INTEGER DEFAULT 0,
  created_at TEXT DEFAULT (datetime('now')),
  updated_at TEXT DEFAULT (datetime('now'))
);

-- 診察記録
CREATE TABLE medical_records (
  id TEXT PRIMARY KEY,
  reservation_id TEXT NOT NULL REFERENCES reservations(id),
  doctor_id TEXT NOT NULL REFERENCES doctors(id),
  patient_id TEXT NOT NULL REFERENCES patients(id),
  diagnosis TEXT,
  prescription TEXT,
  notes TEXT,
  document_issued INTEGER DEFAULT 0,
  document_key TEXT,  -- R2 object key for 診断書PDF
  created_at TEXT DEFAULT (datetime('now'))
);

-- お薬情報
CREATE TABLE prescriptions (
  id TEXT PRIMARY KEY,
  medical_record_id TEXT NOT NULL REFERENCES medical_records(id),
  patient_id TEXT NOT NULL REFERENCES patients(id),
  medication_name TEXT NOT NULL,
  dosage TEXT,
  days INTEGER,
  shipping_status TEXT DEFAULT 'pending',  -- 'pending' | 'shipped' | 'delivered'
  tracking_number TEXT,
  created_at TEXT DEFAULT (datetime('now'))
);
```

## 9. LINE Harness 配信設計

### 9.1 自動配信シナリオ

| トリガー | 配信内容 | タイミング |
|---------|---------|-----------|
| 友だち追加 | ウェルカムメッセージ + ニーズ選択 | 即時 |
| 予約確定 | 予約確認メッセージ（医師名・日時・料金目安） | 即時 |
| 診察前リマインド | 「まもなく診察開始です」+ ビデオ通話リンク | 30分前、5分前 |
| 診察完了 | 決済リクエスト + お薬情報 | 診察終了後 |
| 薬発送 | 発送通知 + 追跡番号 | 発送時 |
| 再診促進 | 「次回の診察はいかがですか？」 | 処方日数終了の3日前 |
| 診断書発行 | 電子ファイル送付 | 発行完了時 |

### 9.2 セグメント配信

| セグメント | タグ | 用途 |
|----------|------|------|
| 初診未完了 | `visit:first_incomplete` | 問診途中離脱者へフォロー |
| 再診期限間近 | `prescription:expiring` | 薬切れ防止リマインド |
| カウンセリング利用者 | `type:counseling` | カウンセリング向けコンテンツ |

## 13. LINE内ビデオ通話の実現性まとめ

| 項目 | 結論 |
|------|------|
| LIFF内WebRTC | **可能**。LIFF v2はWebView上で動作し、WebRTC APIに対応 |
| カメラ・マイク | **可能**。ただしiOSでは`liff.permission`での明示的許可が必要 |
| Cloudflare Calls | **推奨**。Workers統合で管理が容易、SFU方式で品質安定 |
| 画面共有(モバイル) | **制約あり**。iOS WebViewでは非対応 → 書類画像送信で代替 |
| Vアバター(LIFF内) | **可能だが重い**。MediaPipe WASMは動作するが低スペック端末で性能懸念 |
| 通話録画 | **サーバ側で可能**。Cloudflare Callsのrecording APIまたはSFU側で録画 |

**総評: LINE内でのテレビ電話は技術的に実現可能。Zoom遷移より大幅にUX改善できる。Vアバターはフェーズ分けして段階実装が現実的。**
